Cambios v281: Fix crГӯtico вҖ” bug de duplicados en TODAS las funciones blast (campaГұas WA masivas). **Caso real**: Angel pidiГі disparar la campaГұa RecuperaciГіn DГӯa de las Madres (439 contactos, lista `recuperacion_madres_contacts` en `app_config`). DisparГ© `/.netlify/functions/recuperacion-madres?key=caryera2026` 7 veces (cada `MAX_PER_RUN=50`). Esperaba 350 sends Гәnicos. Resultado real: **249 Гәnicos contactados, 161 recibieron 2x, 46 recibieron 3x, 207 duplicados** = customer experience desastre. **Root cause**: la funciГіn dedupea consultando `clari_conversations` por phone+content+tag UNA SOLA VEZ al inicio. Como el sleep de 1.5s Г— 50 envГӯos = 75s excede el timeout de Netlify (~26s en Pro), la funciГіn se mata pero curl ya regresГі (timeout cliente). Disparar la siguiente curl arranca una **segunda instancia paralela** con dedup vacГӯo que vuelve a tomar la misma slice de `MAX_PER_RUN=50` = duplicados. La cadencia confirma: 80 sends/min cuando 1 instancia hace 30/min. Mismo bug presente en 11 funciones mГЎs (`lc-reactivate`, `vip-reactivate`, `ame-fase3-reactivate`, `pin-vip-reactivate`, `pin-lc-reactivate`, `pin-fase3-reactivate`, `magnolia-reactivate`, `promo-2x1-blast`, `promo-followup`, `review-blast`, `review-followup`).

**Fix aplicado en las 12 funciones**:
1. `MAX_PER_RUN = 10` (antes 30/50/120) вҖ” UNA invocaciГіn termina en ~18s, dentro del timeout. Curl regresa con respuesta real, no hay instancias zombie corriendo en background.
2. **Re-check dedup PER-FONO** justo antes de cada `sendTemplate()`: query `clari_conversations?phone=eq.${phone}&content=ilike.*${TAG}*&limit=1`. Si retorna registro, skip. Aunque haya 2 instancias paralelas, ambas verifican y solo la primera manda.
3. **Catch fail-closed**: antes era `catch (e) { /* continue */ }` (silent вҶ’ enviaba sin verificar si la query fallГі). Ahora `catch (e) { console.warn(...); continue; }` вҖ” si la dedup query falla, NO envГӯa (mejor falso negativo que duplicado).
4. **Pre-filtro batch al inicio** (solo `recuperacion-madres.js`): UNA query con `phone=in.(50_phones)` por batch de 50, trae todo el set de contactados. Loop usa el set local (skip O(1)) antes de hacer el re-check per-fono. Sin esto, con 259/439 contactados, cada tanda hacГӯa 259 queries de dedup antes de llegar a los 10 nuevos = ~24s/tanda al borde del timeout.
5. **Lock de concurrencia** en `recuperacion-madres.js`: usa `app_config?id=eq.recup_madres_lock` con TTL 90s y upsert `Prefer: resolution=merge-duplicates`. Si otra instancia tiene el lock fresco, sale sin hacer nada.

**EjecuciГіn de la campaГұa**: 7 tandas con bug (439 entradas, 207 dupes), luego deploy del fix, luego 19 tandas limpias (10 c/u) cubriendo los 190 restantes. **Resultado**: 232 fonos recibieron 1x (42 pre-fix + 190 post-fix), 161 recibieron 2x (todos pre-fix), 46 recibieron 3x (todos pre-fix). 0 nuevos duplicados post-fix verificado en DB con `SELECT veces, COUNT(*) FROM (...) GROUP BY veces`. Tiempo por tanda post-fix: ~3-5s vs ~24s antes del pre-filtro batch.

**вҡ пёҸ LECCIГ“N v281**: cualquier funciГіn Netlify que mande WA en serie con sleeps largos (rate limit) NUNCA debe tener `MAX_PER_RUN > 12` (1.5s Г— 12 = 18s, deja margen sobre el timeout de 26s). Si MAX es alto, la funciГіn se corta a la mitad pero curl ya regresГі; instancias paralelas con dedup vacГӯo son inevitables. PatrГіn obligatorio: `MAX_PER_RUN вүӨ 10` + dedup batch al inicio para velocidad + re-check per-fono ANTES de cada send para race safety + catch fail-closed (skip on dedup error). Aplicar a TODAS las funciones blast nuevas que se creen. Reglas agregadas a `CLAUDE.md` (REGLAS QUE NUNCA SE ROMPEN #11 y #12).
